# 案E: キャラ寮生活シミュレーション「うちの寮」

## 一言で言うと
> 友達2-5人で「寮」を作る。それぞれが自分のAIキャラを持っていて、キャラ同士が寮で共同生活する。毎日ストーリーが自動生成されて、キャラたちの日常ドラマが届く。

## コンセプト

### 体験の流れ
1. 友達2-5人で「寮」を作成
2. 各自がAIキャラを作成（見た目、性格、趣味を設定）
3. キャラたちが寮で勝手に生活を始める
4. ユーザーの現実世界の行動（天気、時間帯、歩数、曜日等）がキャラの行動に影響
5. 毎日「寮の日報」が届く：キャラ同士のドラマが物語として展開
6. たまに「選択肢イベント」が発生 → 全員の選択でストーリーが分岐

### 寮の日報の例
```
🏠 うちの寮 Day 23

朝: りなのコ「モカ」が早起きして朝ごはん作ってた
    ゆきのコ「シロ」はまだ寝てる...（いつものこと）

昼: あいりのコ「ルナ」が新しい服買ってきて
    モカとファッションショーしてたw
    シロは起きてきたけど「眩しい...」って言ってまた寝た

夜: 🔥 イベント発生！🔥
    「寮に猫が迷い込んできた！どうする？」
    → りな: 飼う！
    → ゆき: まだ未回答
    → あいり: まだ未回答
    ※ 全員の回答でストーリーが変わるよ！
```

## ターゲットに刺さる理由

### 「共同生活もの」への憧れ
- テラスハウス、シェアハウス系コンテンツの人気
- **自分たちのキャラで"テラハごっこ"ができる**
- 友達のキャラとの予想外の絡みが面白い
- 実際には一緒に住めない友達との「もしも」体験

### 受動的エンタメの快適さ
- 自分が何かしなくても、**毎日ストーリーが届く**
- 漫画を読む感覚で楽しめる
- でも選択肢イベントで「参加してる感」もある
- **「何もしなくても面白い」が最強の継続率設計**

### キャラの関係性
- キャラ同士に「仲良し度」「ライバル度」などのパラメータ
- 予想外の関係性が生まれる（推し活的な楽しみ）
- **「うちのコとあなたのコ、めっちゃ仲良くなってるんだけどｗ」**

## バイラル設計

### 自然拡散ポイント
1. **寮の日報シェア**: 面白いストーリー展開をスクショ
2. **イベント結果**: 「うちの寮でこんなことが起きた！」
3. **キャラの関係性**: 「うちのコたちの相関図やばい」
4. **新しい寮への招待**: 「一緒に寮作ろう！」

### 二次創作的拡散
- ユーザーがキャラの関係性をイラスト化
- 寮での出来事を漫画にする人が出てくる
- **ユーザー自身がコンテンツクリエイターになる**

## 少人数での楽しみ方
- **2人**: 2人の寮。キャラの掛け合いがメイン
- **3-5人**: 最適人数。ドラマが複雑になって面白い
- **1人**: 「一人暮らし」モード。NPCキャラが同居人として登場

## IP展開の可能性
| 展開 | 内容 |
|------|------|
| ストーリー集 | 人気の寮のエピソードを書籍化 |
| アニメ | ユーザーの寮をショートアニメ化 |
| コラボ寮 | 有名人/ブランドのキャラが期間限定で寮に来る |
| リアルイベント | 「寮祭」リアルイベント |

## 技術要件（概算）
- **LLM API**: ストーリー生成エンジン（複数キャラの関係性を考慮）
- **シミュレーションエンジン**: キャラの行動・関係性計算
- **バックエンド**: グループ管理、ストーリー配信、選択肢管理
- **フロントエンド**: iOS/Androidアプリ
- **イラスト生成/テンプレート**: シーンに合ったビジュアル生成
- **MVP開発期間**: 4-5ヶ月（ストーリー生成の品質確保が鍵）

## リスク
- ストーリー生成のクオリティがサービスの生命線
- LLM APIコストが高い（複数キャラの複雑なストーリー生成）
- グループ作成のハードル（最低2人必要）
- ストーリーのマンネリ化
- テキスト中心になりがち（ビジュアルの補強が必要）
